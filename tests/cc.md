模組化角色與技能 (Modularization) [00:12:16]


針對 AutoAgent-TW (aa-TW) 的優化，若要將其從一個本地開發工具提升到具備如 Claude Code (CC) 般穩定、高靈活度的 Agent 框架，可以從「資源管理」、「狀態持久化」與「系統模組化」三個維度來制定升級計畫。

以下是結合 CC 架構哲學的 aa-TW 具體優化方案：

### 階段一：極致的資源與記憶體管理 (Resource & Memory Optimization)
在處理本地端運算時，記憶體佔用是決定 Agent 流暢度的關鍵。
*   **漸進式載入 (Progressive Disclosure)：** 借鑑 CC 的工具註冊表設計，不要在 aa-TW 啟動時載入所有模組。例如在進行 Python 語法分析時，系統可以根據當前任務的複雜度，動態決定要喚醒哪一個類型檢查器。當任務只需輕量掃描時，僅載入 Ty；遇到需要深層結構分析時，再無縫切換並載入 Pyrefly，用完即釋放，將峰值記憶體壓到最低。
*   **自動上下文瘦身 (Auto-Compression)：** 建立記憶壓縮機制。當本地開發的對話或掃描日誌超過一定 Token 數量時，自動觸發摘要 (Summarize) 而非截斷 (Truncate)，確保系統保留架構脈絡，同時降低本地引擎的負載。

### 階段二：狀態持久化與工作流分離 (State Resilience & Workflow)
本地端開發最怕環境崩潰或終端機意外關閉導致進度遺失。
*   **獨立的工作流狀態 (Workflow State) 快照：** 將「對話狀態」與「執行狀態」徹底分離。系統應將每一步操作（例如：已完成的程式碼重構、已掃描的檔案清單）寫入本地的 JSON 狀態檔中，並賦予唯一編號 (Checkpointing)。
*   **無縫接軌恢復 (Resume Capability)：** 若本地引擎重啟或斷線，aa-TW 能直接讀取狀態檔，精準回到上一次中斷的步驟，避免重複執行已完成的靜態分析或型別檢查，浪費運算資源。

### 階段三：多層安全與工具防禦 (Defense in Depth)
即使在本地運行，也需要防止 Agent 誤刪檔案或執行錯誤腳本。
*   **指令攔截與沙箱機制：** 建立類似 CC 的防禦縱深。對於一般的讀取操作 (如 `git status` 或讀取檔案) 可以直接放行；但對於會修改系統、覆寫檔案的破壞性操作，必須有明確的權限判定，甚至在終端機跳出確認提示，確保「Human in the Loop」。

### 階段四：CLI 可觀測性與系統日誌 (Observability)
為了更精準地掌握 aa-TW 在本地的運作效能，需要提升系統的透明度。
*   **串流狀態廣播 (Streaming Events)：** 在 CLI 介面中，除了輸出文字結果，也應該即時顯示系統狀態（例如：`[切換至 Pyrefly 模式]`、`[記憶體優化中...]`、`[工具載入中...]`）。
*   **結構化系統日誌 (System Logging)：** 將對話日誌與系統執行日誌分開。系統日誌應詳細記錄每一次類型檢查器的觸發時間、耗用記憶體變化、工具呼叫的成功與失敗率，作為後續除錯與效能調校的核心數據。

### 階段五：技能模組化 (Skill Modularization)
*   **可插拔技能庫：** 將特定任務封裝成獨立的 Skill 模組。例如建立一個「記憶體分析 Skill」或「架構重構 Skill」，每個 Skill 自帶專屬的 Prompt 模板與所需的輸入/輸出格式定義。未來要擴充 aa-TW 的能力時，只需新增 Skill 模組，完全不需要改動核心框架。
