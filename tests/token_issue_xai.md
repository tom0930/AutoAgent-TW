### a. `handoff.json` 是做什麼的？

**handoff.json** 是 **斷點續傳 / 狀態交接** 檔案（State Handoff File）。

它的作用就像「存檔點」：

- 當 Agent 處理完一個大目錄（例如 `z:\02` 全部完成）後，把目前進度、已處理的波次（wave）、結果摘要、剩餘待辦等關鍵狀態**寫入** `handoff.json`。
- 下次重新開新對話或新 Session 時，Agent 先讀取這個 `handoff.json`，就能知道「之前做到哪裡了」，不需要從頭再跑一次。
- 好處：避免在同一個超長對話中累積 token，也大幅降低「重開 IDE 才能正常」的依賴。

簡單說：它是讓長任務能**分段執行**的安全橋樑。

### b. 「超出 LLM Context 限制 或 IDE 外掛記憶體過重」該怎麼解決？

這是這次 bug 的**根本原因**，有幾個實用解法（由易到難）：

1. **最立即有效**：強制定期重置上下文
   - 每處理完 1 個目錄（或每完成 3–5 個 wave）就主動告訴使用者：「請關閉目前對話，新開一個對話繼續」，並附上 `handoff.json`。

2. **減少對話內輸出量**
   - 不要把每一張發票的完整明細都印在聊天視窗。
   - 改成只輸出摘要 + 把詳細結果寫入 JSON 檔案。
   - 大量 `view_file` 呼叫後，適時呼叫清理歷史或 summarization。

3. **技術層面優化**
   - 使用較長 Context 的模型（如果平台允許）。
   - IDE 外掛端增加自動截斷舊歷史的功能。
   - 關鍵操作（尤其是檔案清理）**改用 Python 腳本** 而非 Shell 指令。

4. **長期解**：引入 **Handoff 機制**（就是方案 A），讓 Agent 自動管理狀態，不依賴單一長對話。
